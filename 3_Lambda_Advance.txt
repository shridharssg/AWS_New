============================================================
✅ 1. EXECUTION ENVIRONMENT
------------------------------------------------------------

Lambda runs inside execution environment:

• Runtime
• Code
• Dependencies

✔ Created on demand
✔ Reused sometimes
✔ Destroyed after idle

👉 This leads to cold start / warm start


============================================================
	✅ 1.1 LAMBDA EXECUTION LIFECYCLE
	------------------------------------------------------------

	Every Lambda execution has 2 phases:

	1. INIT PHASE (Cold start only)
	   • Runtime initialization
	   • Code loading
	   • Dependency loading
	   • Global code execution

	2. INVOKE PHASE
	   • Handler function executes

	------------------------------------------------------------

	Flow:

	Request → INIT → INVOKE → Response

	------------------------------------------------------------

	Important:

	✔ INIT runs only in cold start
	✔ Warm start skips INIT
	✔ INIT time adds latency

============================================================
✅ 2. COLD START vs WARM START
------------------------------------------------------------

Cold Start:

AWS must:
• Create environment
• Load runtime
• Load code
• Initialize dependencies

⏱ Delay: 100ms – 1000ms+

------------------------------------------------------------

When happens:

✅ First request
✅ After inactivity
✅ Scaling (new instances)
✅ Deployment change
✅ VPC enabled

------------------------------------------------------------

Warm Start:

✔ Existing environment reused
✔ Only handler runs
✔ Much faster

------------------------------------------------------------

How to reduce cold start:

✔ Keep code small
✔ Avoid heavy libraries
✔ Move init outside handler
✔ Avoid VPC if possible
✔ Use Provisioned Concurrency

✅ 3. SCALING
------------------------------------------------------------

🔹 What is scaling?

Scaling = Lambda automatically increases instances based on incoming requests

Example:
1 request → 1 instance
1000 requests → 1000 instances



✅ Case 1: First 10 requests (cold start)
	Time = T0
	10 requests arrive at same time

	👉 AWS creates:
	10 NEW execution environments

	✅ All are cold start

✅ Case 2: Next 10 requests (after 1 second)

	Previous 10 environments are FREE (if execution completed)

	So:
	New 10 requests → reuse SAME 10 environments

	✅ This is warm start
------------------------------------------------------------

🔹 How scaling works internally:

	• Each request = new execution environment
	• No shared threads
	• Horizontal scaling

------------------------------------------------------------

🔹 Default scaling limit:

✔ 1000 concurrent executions per region

------------------------------------------------------------

🔹 Scaling scenarios:

	✅ Traffic spike → Lambda creates more instances
	✅ Low traffic → instances removed

	👉 Scale up + scale down automatically

------------------------------------------------------------

🔹 SCALING PROBLEM (IMPORTANT)

❌ Over-scaling problems:

• Too many DB connections
• Downstream system crash
• Cost spike

Example:
1000 Lambdas → 1000 DB connections → DB crash

------------------------------------------------------------

🔹 Under-scaling problems:

❌ Throttling
❌ Request failure (429 error)

------------------------------------------------------------

🔹 CONTROLS FOR SCALING

✅ Reserved Concurrency

Example:
Limit Lambda to 50

✔ Protect DB
✔ Prevent overload

✅ Provisioned Concurrency 
	Pre-warms Lambda instances so they are always ready (no cold start)
	default Provisioned Concurrency = 0

Pre-warm instances:

✔ No cold start
❌ Extra cost (even if unused)

is used to reduce cold start but extra billing


	✅✅ HOW TO VERIFY (IMPORTANT)
		Go to:	👉 Monitoring → CloudWatch

		Check metrics:

			ProvisionedConcurrentExecutions
			ProvisionedConcurrencyUtilization
			
 You CANNOT enable it directly on function
	👉 You must use:
	Version + Alias (mandatory)			
				
	Step 1: Create version
	Step 2: Create alias
	Step 3: Enable provisioned concurrency
	Step 4: Set number (e.g. 5)			
------------------------------------------------------------

🔹 Over-provisioning problem:

❌ Paying for unused resources

Example:
Provisioned = 100
Actual usage = 10

👉 90 instances wasted → cost issue

------------------------------------------------------------

🔹 Interview Answer:

“Lambda auto-scales per request, but we must control scaling
using reserved concurrency to protect downstream systems.”

✅ 9. SYNC vs ASYNC INVOCATION (CRITICAL CONCEPT)
------------------------------------------------------------

	🔹 What is invocation?

	Invocation = how Lambda is triggered and how it responds

	------------------------------------------------------------

	✅ 1. SYNCHRONOUS INVOCATION (SYNC)

	Definition:
	Caller waits for Lambda response

	Flow:
	Client → API Gateway → Lambda → Response → Client

	------------------------------------------------------------

	Characteristics:

	✔ Client waits for response  
	✔ Single request-response cycle  
	✔ No automatic retries  
	✔ Timeout directly affects user  

	------------------------------------------------------------

	Examples:

	• API Gateway (REST API)
	• ALB (Application Load Balancer)

	------------------------------------------------------------

	Failure behavior:

	❌ If Lambda fails → error returned immediately  
	❌ No retry by AWS  

	👉 Client must handle retry

	------------------------------------------------------------

	Real Problem:

	If API Lambda is slow:
	❌ User waits  
	❌ Timeout  
	❌ Bad UX  

	------------------------------------------------------------


	✅ 2. ASYNCHRONOUS INVOCATION (ASYNC)

	Definition:
	Caller does NOT wait for response

	Flow:
	Event → Lambda → Background processing

	------------------------------------------------------------

	Characteristics:

	✔ Fire-and-forget  
	✔ Event stored internally  
	✔ Lambda runs separately  
	✔ Automatic retries  

	------------------------------------------------------------

	Examples:

	• SQS
	• SNS
	• S3 events
	• EventBridge

	------------------------------------------------------------

	Failure behavior:

	✔ AWS retries automatically (2+ times)  
	✔ If still fails → goes to DLQ  

	------------------------------------------------------------

	Real Problem:

	Retries can cause:
	❌ Duplicate operations  
	❌ Double payment / email  

	👉 Must handle idempotency

	------------------------------------------------------------


	✅ 3. WHEN TO USE SYNC vs ASYNC

	Use SYNC:

	✔ User needs immediate response  
	✔ Login / Signup  
	✔ Get data  

	------------------------------------------------------------

	Use ASYNC:

	✔ Heavy processing  
	✔ Background jobs  
	✔ Payment / Email / Notifications  

	------------------------------------------------------------


	✅ 4. HYBRID REAL-WORLD FLOW (IMPORTANT)

	Best practice:

	Client
	 → API Gateway
	   → API Lambda (SYNC)
		  ✔ validate
		  ✔ save request
		  ✔ send to SQS
		  ✔ return response

	SQS
	 → Worker Lambda (ASYNC)
		  ✔ process background tasks

	“Lambda supports synchronous and asynchronous invocation.
	Synchronous is used for user-facing APIs with no retries,
	while asynchronous is used for background processing with retries and DLQ handling.”

✅ 10 EVENT SOURCE MAPPING (VERY IMPORTANT)
	------------------------------------------------------------

	Lambda does NOT directly read from SQS.

	AWS creates a poller:

	SQS → Event Source Mapping → Lambda

	------------------------------------------------------------

	Responsibilities:

	✔ Poll messages from queue
	✔ Batch messages
	✔ Invoke Lambda

	------------------------------------------------------------

	Important behavior:

	✔ Default batch size: 10
	✔ Lambda processes messages in batches
	✔ Retry happens per batch

	------------------------------------------------------------

	Problem:

	If 1 message fails:
	❌ Entire batch retried

	------------------------------------------------------------

	Solution:

	✔ Partial batch response (advanced)

	============================================================
	
	
✅ 11 SQS VISIBILITY TIMEOUT (CRITICAL)
	------------------------------------------------------------

	When Lambda reads a message:

	✔ Message becomes invisible temporarily

	------------------------------------------------------------

	If Lambda succeeds:

	✔ Message is deleted

	If Lambda fails:

	✔ Message becomes visible again → retry

	------------------------------------------------------------

	Important rule:

	Visibility timeout > Lambda execution time

	------------------------------------------------------------

	Problem example:

	Lambda time = 10 sec  
	Visibility = 5 sec  

	❌ Message processed again (duplicate)

	============================================================

	============================================================
✅ 12 BATCH PROCESSING & DUPLICATES
	------------------------------------------------------------

	Lambda processes SQS in batches:

	Example:
	Batch size = 10

	------------------------------------------------------------

	If 1 fails:

	❌ Entire batch retried

	------------------------------------------------------------

	Result:

	✔ Duplicate processing
	✔ Already successful messages re-run

	------------------------------------------------------------

	Solution:

	✔ Idempotency
	✔ Partial batch response

✅ 13. TIMEOUT ISSUE

	✅  API GATEWAY vs LAMBDA TIMEOUT
	------------------------------------------------------------

	API Gateway timeout = 29 seconds (max)

	Lambda timeout = up to 15 minutes

	------------------------------------------------------------

	Problem:

	❌ API will fail after 29 sec
	even if Lambda continues running

	------------------------------------------------------------

	Impact:

	✔ Client gets timeout error
	✔ Lambda may still be running in background

	------------------------------------------------------------

	Solution:

	✔ Use SQS for long processing
	✔ Return response early
