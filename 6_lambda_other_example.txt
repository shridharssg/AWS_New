✅ 1. REAL THROTTLING HANDLING (STEP BY STEP)
------------------------------------------------------------

🎯 Goal:
Identify and fix Lambda throttling in production

------------------------------------------------------------

✅ What is throttling?

• Happens when concurrency limit is reached
• Lambda cannot create more instances
• Requests fail with 429 error

------------------------------------------------------------

✅ How to DETECT throttling (AWS Console):

1. Go to AWS Console
2. Open CloudWatch
3. Click "Metrics"
4. Select "Lambda"
5. Choose your function

Check metrics:
✔ Throttles > 0 → throttling happening
✔ ConcurrentExecutions → reached limit [1](https://docs.aws.amazon.com/lambda/latest/dg/troubleshooting-event-source-mapping.html)  

------------------------------------------------------------

✅ How to FIX throttling:

✔ Option 1: Increase concurrency limit  
✔ Option 2: Reduce execution time (optimize code)  
✔ Option 3: Increase batch size (SQS use case)  
✔ Option 4: Use queue (SQS) instead of direct API  

------------------------------------------------------------

✅ Real Example:

Traffic = 200 req/sec  
Limit = 50  

→ 150 requests throttled ❌  
→ Use SQS + worker Lambda ✅

✅ 2. DLQ INSPECTION & REPROCESSING (STEP BY STEP)
------------------------------------------------------------

🎯 Goal:
Handle failed messages safely

------------------------------------------------------------

✅ What is DLQ?

• Stores messages that failed after retries
• Prevents data loss [2](https://oneuptime.com/blog/post/2026-02-12-configure-dead-letter-queues-lambda-functions/view)  

------------------------------------------------------------

✅ How to CHECK DLQ:

1. Go to AWS Console
2. Open "SQS"
3. Select your DLQ (dead-letter queue)
4. Check:
   ✔ Messages available > 0 → failures exist
5. Click "Send and receive messages"
6. View message content

------------------------------------------------------------

✅ How to REPROCESS messages:

✅ Option 1 (Manual):

1. Open DLQ
2. Select message
3. Click "Start DLQ redrive"
4. Move message back to main queue

------------------------------------------------------------

✅ Option 2 (Recommended):

✔ Create Lambda to reprocess DLQ
✔ Trigger manually when needed

------------------------------------------------------------

✅ Important:

• Always check WHY message failed first
• Fix issue before reprocessing
