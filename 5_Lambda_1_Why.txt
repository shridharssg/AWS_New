✔ POST /auth/login
   • Validate email/password

   • Fetch product list
   • Sorting, filtering

✔ POST /orders/create
   • Create order
   • Deduct inventory
   • Send email

✅ Covers:
• Sync APIs
• DB calls
• Heavy processing
• External integrations

✅ 2. NODE.JS EXPRESS ARCHITECTURE (CURRENT MODEL)
------------------------------------------------------------

Basic structure:

const express = require("express");
const app = express();

app.post("/auth/login", loginController);
app.get("/products/list", productController);
app.post("/orders/create", orderController);

app.listen(3000);

✅ Deployment:
• EC2
• ECS / Docker
• On-prem

✔ This is server-based architecture


============================================================
✅ 3. EXPRESS DRAWBACKS (CRITICAL FOR INTERVIEWS)
------------------------------------------------------------

✅ 1. Cost Problem
--------------------------------
• Server runs 24×7
• Cost even if no traffic

Example:
• EC2 = ₹2500/month minimum

❌ Waste of money


✅ 2. Server Maintenance Required
--------------------------------
• OS patching
• Security updates
• Scaling configs
• Load balancer setup

❌ Full responsibility on you


✅ 3. Scaling is Manual
--------------------------------
If sudden traffic spike:

• Add servers
• Configure autoscaling
• Manage load balancing

❌ Slow and risky
❌ Downtime possible


✅ 4. Single Point Failure
--------------------------------
• One crash → entire app down

Example:
• Memory leak → all APIs fail

❌ Poor reliability


✅ 5. Over-Provisioning
--------------------------------
• Large server for rare traffic spikes

Example:
• Peak only 1 hr/day
• Server runs full day

❌ 90% resource wasted


✅ 6. Monolithic Architecture
--------------------------------
All services inside one app:

• Auth
• Product
• Orders

❌ Hard to scale individually
❌ Risky deployments


✅ 7. Slow Deployment
--------------------------------
• Whole app redeploy
• Downtime possible


✅ 8. Heavy Task Blocking
--------------------------------
Example:
• PDF generation in order API

❌ Blocks Node event loop
❌ Slows all APIs


✅ 9. Security Burden
--------------------------------
• Firewall
• SSL
• Ports

❌ Fully managed by you


✅ 10. Multi-Region Hard
--------------------------------
Need:
• Multiple servers
• Sync deployments

❌ Complex setup


============================================================
✅ 4. HOW AWS LAMBDA SOLVES THESE
------------------------------------------------------------

✅ 1. Pay-per-use (Biggest advantage)
--------------------------------
• Pay only when code runs
• No cost when idle

➡️ You pay only for execution time in milliseconds [1](https://aws.amazon.com/lambda/pricing/)


✅ 2. No Server Management
--------------------------------
• No EC2
• No patching
• No scaling infra

✔ AWS handles everything


✅ 3. Auto Scaling
--------------------------------
• Automatically scales to thousands of requests

✔ No manual config needed [2](https://www.geeksforgeeks.org/devops/introduction-to-aws-lambda/)


✅ 4. Isolation (Microservices)
--------------------------------
Each API = separate Lambda

✔ Failure does not affect others


✅ 5. No Over-Provisioning
--------------------------------
• Runs only when triggered

✔ Zero idle cost


✅ 6. Modular Architecture
--------------------------------
• Each function independent

✔ Easier scaling and maintenance


✅ 7. Fast Deployment
--------------------------------
• Deploy one function at a time

✔ Faster CI/CD


✅ 8. Heavy Tasks Handled Separately
--------------------------------
• Run heavy tasks in separate Lambda

✔ No blocking of main API


✅ 9. Security Managed by AWS
--------------------------------
• IAM-based access control
• No direct server exposure


✅ 10. Multi-Region Easy
--------------------------------
• Deploy Lambda in multiple regions easily

✔ API Gateway handles routing


============================================================
✅ 5. EVENT-DRIVEN EXECUTION MODEL (IMPORTANT)
------------------------------------------------------------

Flow:

Trigger → Lambda → Output

Example:

• API call → Lambda runs
• S3 upload → Lambda resizes image

👉 Lambda runs only on event trigger and stays idle otherwise [2](https://www.geeksforgeeks.org/devops/introduction-to-aws-lambda/)

------------------------------------------------------------

✔ /auth/login
   Express → server cost always
   Lambda → pay-per-use

✔ /products/list
   Express → scaling issue
   Lambda → auto-scale

✔ /orders/create

   Lambda → async processing


============================================================
✅ 7. WHEN NOT TO USE LAMBDA
------------------------------------------------------------

❌ Long-running tasks (15+ min)
❌ Constant traffic (EC2 cheaper)
❌ Real-time low-latency critical apps


============================================================
✅ 8. FINAL INTERVIEW ANSWER
------------------------------------------------------------

“Lambda is used to remove server management,
enable auto-scaling, and follow pay-per-use model.

Unlike Express where servers run continuously,
Lambda runs only on demand, reducing cost and improving scalability.”
